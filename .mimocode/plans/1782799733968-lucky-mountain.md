# 社区板块功能实现计划（含合规要求）

## 目标
实现基础社区功能：能发帖、能看帖、能点赞，同时满足中国网络安全法规要求。

---

## 1. 数据库建表

### community_posts（帖子表）
```sql
CREATE TABLE community_posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  profile_id UUID REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  content TEXT NOT NULL,
  images TEXT[] DEFAULT '{}',
  pet_type TEXT,
  breed TEXT,
  likes_count INT DEFAULT 0,
  comments_count INT DEFAULT 0,
  review_status TEXT DEFAULT 'pending',  -- pending | approved | rejected
  reject_reason TEXT,
  is_deleted BOOLEAN DEFAULT FALSE,     -- 软删除
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### community_likes（点赞表）
```sql
CREATE TABLE community_likes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id UUID REFERENCES community_posts(id) ON DELETE CASCADE NOT NULL,
  profile_id UUID REFERENCES profiles(id) ON DELETE CASCADE NOT NULL,
  created_at TIMESTAMPTZ DEFAULT now(),
  UNIQUE(post_id, profile_id)
);
```

### community_reports（举报表）
```sql
CREATE TABLE community_reports (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id UUID REFERENCES community_posts(id) ON DELETE CASCADE NOT NULL,
  reporter_id UUID REFERENCES profiles(id) ON DELETE SET NULL,
  reason TEXT NOT NULL,        -- 举报原因
  category TEXT NOT NULL,      -- spam | violence |色情 | political | other
  status TEXT DEFAULT 'pending', -- pending | reviewed | resolved
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### content_audit_logs（审核日志表，留存6个月）
```sql
CREATE TABLE content_audit_logs (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  content_type TEXT NOT NULL,  -- post | comment
  content_id UUID NOT NULL,
  profile_id UUID REFERENCES profiles(id),
  content_text TEXT,
  content_images TEXT[],
  audit_source TEXT NOT NULL,  -- keyword_filter | api审核 | 人工审核
  audit_result TEXT NOT NULL,  -- pass | reject
  reject_reason TEXT,
  ip_address TEXT,
  created_at TIMESTAMPTZ DEFAULT now()
);
```

### RLS 策略
- 帖子：所有人可读，作者可写，软删除
- 点赞：所有人可读，用户只能操作自己的
- 举报：所有人可插入，只有管理员可读取和更新状态
- 审核日志：仅管理员可读

### 索引
```sql
CREATE INDEX idx_posts_created ON community_posts(created_at DESC) WHERE is_deleted = FALSE;
CREATE INDEX idx_posts_breed ON community_posts(breed) WHERE is_deleted = FALSE;
CREATE INDEX idx_posts_review ON community_posts(review_status);
CREATE INDEX idx_likes_post ON community_likes(post_id);
CREATE INDEX idx_likes_profile ON community_likes(profile_id);
CREATE INDEX idx_reports_status ON community_reports(status);
CREATE INDEX idx_audit_created ON content_audit_logs(created_at DESC);
```

## 2. 合规功能模块

### 2.1 内容安全审核（网络安全法第12条）

**方案：前端关键词过滤 + 后端 API 审核（双重机制）**

#### 2.1.1 本地敏感词过滤
**文件**: `src/lib/content-filter.js`

```js
功能：
- 敏感词库（内置基础词库 + 可从后端拉取扩展词库）
- 发布前前端预检：文字内容过一遍本地词库
- 匹配到敏感词时返回：敏感词位置、建议替换词
- 仅作为前端预检，最终审核以后端为准
```

词库管理：
- 基础敏感词库内置在前端（约 500-1000 词）
- 支持从后端动态拉取扩展词库
- 词库格式：`{ word: string, level: 'block' | 'review', replacement: string }`
- block 级别：直接拦截，不允许发布
- review 级别：发布后进入人工审核队列

#### 2.1.2 服务端审核 API 对接
**文件**: `src/lib/content-audit.js`

```js
功能：
- 发布帖子后调用 /api/content/audit 接口
- 接口对接第三方审核服务（网易易盾/阿里云内容安全）
- 审核结果：pass（通过）| reject（拒绝）| pending（待人工审核）
- 被拒绝的帖子显示拒绝原因
- 待审核的帖子不公开显示，仅作者可见
```

审核流程：
```
用户发布 → 本地敏感词预检 → 插入DB(review_status=pending)
  → 异步调用审核API → 审核通过 → review_status=approved → 公开显示
                        → 审核拒绝 → review_status=rejected → 通知用户
```

### 2.2 举报功能（互联网信息服务管理办法第15条）

**文件**: `src/views/Community.vue`（举报弹窗）

```js
功能：
- 帖子右上角增加举报按钮（三个点菜单）
- 举报原因选择：垃圾信息 | 暴力内容 | 色情内容 | 政治敏感 | 其他
- 可选填写补充说明
- 同一用户对同一帖子只能举报一次
- 举报记录存入 community_reports 表
- 管理后台可查看举报列表并处理
```

### 2.3 用户协议与社区规范

**文件**: `src/views/Community.vue`（发帖前确认）

```js
功能：
- 首次发帖前弹窗展示《用户协议》和《社区规范》
- 用户必须勾选"已阅读并同意"才能发布
- 协议内容存储在 localStorage，仅弹一次
- 协议要点：
  - 禁止发布违法违规内容
  - 禁止发布虚假信息
  - 禁止发布广告引流内容
  - 用户对发布内容承担法律责任
  - 平台有权删除违规内容
```

### 2.4 数据留存（网络安全法第21条）

**实现方式**：
- 帖子使用软删除（`is_deleted` 字段），删除后数据仍保留
- 发布日志自动记录：用户ID、内容、时间、IP地址
- 审核日志表 `content_audit_logs` 自动记录所有审核操作
- Supabase 数据保留策略：至少 6 个月
- 建议在 Supabase Dashboard 设置数据保留策略

### 2.5 用户数据权利（个人信息保护法）

**实现方式**：
- 用户可删除自己的帖子（软删除，数据保留）
- 用户可在个人中心查看发布记录
- 用户注销时需提供删除帖子的功能（当前已有账号注销逻辑）
- 数据导出功能（可选，后续扩展）

### 2.6 未成年人保护

**实现方式**：
- 注册时收集年龄信息（当前系统已有生日字段）
- 14 岁以下用户发帖时弹窗提示需监护人同意
- 不在社区推荐可能影响未成年人身心健康的内容

## 3. 新建 Composable: useCommunity.js

**文件**: `src/composables/useCommunity.js`

```js
功能：
- fetchPosts({ petType, breed, page })  // 分页加载，只加载 approved 的帖子
- createPost({ content, images, petType, breed })  // 发帖（本地预检 + 上传）
- toggleLike(postId)  // 点赞/取消（optimistic UI）
- reportPost(postId, reason, category)  // 举报
- deletePost(postId)  // 删除自己的帖子
- checkSensitiveWords(text)  // 敏感词预检
- uploadImages(files)  // 图片上传到 Supabase Storage
- fetchMyPosts()  // 我的帖子列表
```

核心查询逻辑：
- 帖子列表 JOIN `profiles` 获取 `display_name`、`avatar_url`
- 只查询 `review_status = 'approved'` 且 `is_deleted = FALSE` 的帖子
- 查询 `community_likes` 判断当前用户是否点赞
- 分页用 `range` 实现，每次加载 20 条

┌─────────────────────────┐

### 发帖流程（合规流程）
```
1. 用户点击发帖按钮
2. 首次发帖 → 弹出用户协议确认框
3. 输入文字内容
4. 可选添加图片（最多9张）
5. 选择猫/狗类型和品种
6. 点击发布
7. 前端本地敏感词预检
   - 发现 block 级敏感词 → 提示"包含违规内容，无法发布"
   - 发现 review 级敏感词 → 提示"内容将进入审核"
8. 上传图片到 Supabase Storage
9. 插入 DB（review_status=pending）
10. 异步调用审核 API
11. 审核通过 → 帖子显示在列表
12. 审核拒绝 → 通知用户，帖子不显示
```

### 举报弹窗
- 帖子右上角三个点菜单
- 选择举报原因
- 填写补充说明（可选）
- 提交举报

## 7. 验证方式

1. 运行 SQL migration 创建表
2. 启动 `npm run dev`，登录后进入社区 Tab
3. 验证帖子列表为空，显示空状态
4. 点击发帖按钮，首次发帖弹出用户协议
5. 输入包含敏感词的内容，验证前端拦截
6. 输入正常内容发布，帖子出现在列表（可能有审核延迟）
7. 刷新页面，帖子持久化
8. 点击点赞，计数 +1，再次点击取消
9. 点击举报按钮，提交举报
10. 切换猫/狗筛选，验证过滤生效
